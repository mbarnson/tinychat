# 01-01 Plan — Foundation and macOS local CoreAI inference

Date: 2026-07-02

## Objective

Replace the stock SwiftData template with the smallest real tinychat foundation and prove macOS real Qwen3 0.6B streaming inference inside the beginning of the final chat UI.

This plan intentionally proves macOS first. iOS real inference waits for the download/cache path because physical iOS App Support cache should not be seeded by hacks.

## Context

- `vendor/coreai-models/swift/Sources/CoreAILanguageModels/LanguageModel/CoreAIRunner.swift`
- `vendor/coreai-models/python/src/coreai_models/model_registry.py`

Important source facts:

- CoreAI/CoreAILM is macOS/iOS 27.0+.
- Qwen3 0.6B macOS CoreAI preset context is 8192.
- Qwen3 0.6B iOS CoreAI preset context is 4096.
- FoundationModels wrapper currently maps only temperature from `GenerationOptions`.
- CoreAI wrapper emits reasoning separately and skips prior `.reasoning` when rebuilding prompts.

## Decisions locked by grilling

- App dependency: Apple CoreAI Models GitHub package tracking `main`.
- `vendor/coreai-models` is local reference/export checkout only.
- Use FoundationModels wrapper first; accept temperature-only sampling.
- Thinking defaults on; composer toggle injects hidden `/think` or `/no_think`.
- Thinking temperature: 0.6. Non-thinking temperature: 0.7.
- Store reasoning separately, display collapsibly, copy separately, exclude from future prompts/summaries.
- Persist messages at end/stop, not per token.
- Stop keeps partial assistant output.
- Title generation is prompt-only, non-thinking, after first assistant response; fallback to first prompt if title generation fails.
- Title generation may invalidate/rebuild the initial chat cache.
- Code stays in app target initially.

### Task 3 — Minimal final chat UI with real macOS streaming

Type: implementation

Files likely touched:

- `tinychat/ContentView.swift`
- new view/model/service files under `tinychat/`
- `tinychatUITests/`

Actions:

1. Build the beginning of the final chat UI, not a throwaway spike:

   - multiline composer

   - real CoreAI implementation
   - deterministic test implementation for fast UI tests
3. Implement `CoreAIChatEngine` using `CoreAILanguageModel(resourcesAt:)` and `LanguageModelSession`.
4. Stream response into view state.
5. Persist user and assistant messages at completion/stop.

Verification:

  - send prompt
  - observe streamed response
  - stop generation
  - verify persisted message after relaunch
  - verify copy buttons exist

  - send short deterministic-ish prompt
  - assert assistant text becomes non-empty
  - do not assert exact model prose
- `xcodebuild` macOS tests pass.

Done when:

- macOS app can stream a real Qwen3 0.6B response from local App Support cache.
- Messages persist.
- Reasoning is separated from final answer.
- First chat gets a generated or fallback title.


Use `xcodebuild -showdestinations` if destination names differ.

## Deviation rules

- Fix build/test blockers immediately and document in summary.
- If CoreAI package `main` has API drift, inspect vendored source and adapt without adding shims unless needed for current build.
- If real model export cannot run on this machine, finish all app/test infrastructure and report the exact missing prerequisite.
- Do not add download UI in this plan beyond model-missing/install-present status. Download belongs to Phase 02.
- Do not add iOS real inference in this plan. iOS real inference belongs after download/cache flow.
